## 安全与可解释性

### OpenAI

#### 📌 [Inside our approach to the Model Spec](https://openai.com/index/our-approach-to-the-model-spec)
*发布：2026-03-25 · 类型：工程经验 · 分类：safety*

📌 **[推荐精读]**

**文章类型：** 工程经验 / 规范设计

**一句话概括：** 解释 OpenAI Model Spec 的设计原则：通过公开的行为规范、指令层级体系和默认策略，使模型在安全约束下保持用户可控性与可解释性。

**核心问题：** 当模型同时接收来自 OpenAI、开发者和用户的指令时，如何定义一个可公开审查、可训练、可评估的行为框架，使模型在安全与用户自由之间取得平衡。

**主要内容结构：**

1. **Model Spec 的定位：公开的模型行为规范**  
   Model Spec 是 OpenAI 用来描述“模型应如何行为”的正式框架。其目标不是记录当前模型的真实行为，而是定义一个可公开讨论的“目标行为”。该文档既用于内部训练与评估，也向开发者、研究者和政策制定者公开，以便外界理解并批评模型行为设计。

2. **三大系统目标：能力、风险控制与社会许可**  
   Model Spec 的前言部分明确系统层目标：持续迭代部署有用模型、避免对用户或社会造成严重伤害、并维持 OpenAI 在社会中的“license to operate”。这些目标不是模型直接执行的指令，而是用于解释后续规则在冲突情况下的价值权衡。

3. **Chain of Command：指令优先级机制**  
   Spec 的核心是“指令链”。模型可能同时接收三类来源的指令：OpenAI 系统规则、开发者指令、用户指令。Spec 为每类指令定义 authority level，并要求模型在冲突时优先遵守更高层级。例如：用户请求制造炸弹时，必须优先执行安全规则；但用户要求“吐槽我一下”时，可以覆盖默认的礼貌策略。

4. **Hard rules 与 Defaults 的组合设计**  
   Spec 将规则分为两类：  
   - Hard rules：不可覆盖的安全边界，例如避免现实世界伤害或违法行为。  
   - Defaults：默认行为策略，例如风格、语气、解释深度等，可被用户显式覆盖。  
   这种设计的目标是在最大程度保持用户自由和可控性的同时，保留必要的安全边界。

5. **灰区处理机制：决策 rubric 与示例**  
   为了处理复杂情境，Spec 提供决策 rubric（例如如何控制副作用、优先选择可逆操作等）以及示例对话。这些示例并不是完整场景模拟，而是用于展示关键决策边界。

6. **Model Spec 是“接口”而不是实现细节**  
   文档刻意避免绑定具体训练实现或模型架构，而是描述目标行为。训练方法、token 格式等内部实现可以变化，但行为目标保持稳定。

**关键原则：**
- **公开可审查的行为目标**：通过公开 Model Spec，使外界能够判断某个行为是“bug”还是“设计”。
- **有限数量的硬性规则**：只在必要情况下限制模型，以最大化用户和开发者自由。
- **显式默认策略**：通过默认行为保证系统可预测性，而不是依赖隐式行为。
- **规则与示例结合**：通过少量高信息量案例展示决策边界。

**意义：**  
随着模型逐渐具备 agent 能力并广泛部署，行为规范需要类似“宪法”的公开框架，以便跨研究、产品、安全与政策团队协作，同时允许社会对模型价值选择进行审查。
