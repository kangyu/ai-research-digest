## 多模态

### Google / Hugging Face

#### [Welcome Gemma 3: Google's All New Multimodal, Multilingual, Long Context Open LLM](https://huggingface.co/blog/gemma3)
*发布：2026-03-17 · 分类：`multimodal` · `pretraining`*

**核心贡献：** Google 发布 Gemma 3 开源模型系列（1B/4B/12B/27B），在 Gemma 2 基础上全面升级为多模态、多语言、长上下文能力。其中 Gemma 3 27B-IT 在 LMSys Chatbot Arena Elo 评分达 1339，成为全球前十模型之一，且与 o1-preview 性能相当。

**技术细节：**
- **规模与模态**：1B 为纯文本模型；4B/12B/27B 支持图文混合输入，均集成 SigLIP 视觉编码器。视觉编码器将图像缩放至 896×896 作为固定输入分辨率；为处理非方形或高分辨率图像，引入 **Pan & Scan** 算法——自适应裁剪图像为多个子块，分别编码后合并，有效支持局部细节识别。
- **上下文长度**：1B 版本支持 32k tokens，其余版本支持 128k tokens。长上下文能力通过在预训练末尾阶段以 128k 序列继续训练（非从头训练）实现，节省大量算力。RoPE 位置编码基频从 Gemma 2 的 10k 升级至 1M，并以 8 倍缩放因子适配长上下文。
- **注意力机制**：沿用并改进 Gemma 2 的滑动窗口交叉注意力（sliding window interleaved attention）。局部层与全局层的比例从原来的 1:1 调整为 5:1（5 个局部层 + 1 个全局层），窗口大小从 4096 缩减至 1024 tokens，在不损失 perplexity 的前提下显著降低 KV Cache 内存占用。图像 token 使用双向注意力（无因果掩码），文本 token 使用因果注意力，两者在同一框架内共存。
- **多语言**：预训练数据中多语言数据量翻倍；Tokenizer 升级为与 Gemini 2.0 相同的 SentencePiece，词表扩展至 262k 条目，显著改善中文、日文、韩文的编码效率，支持 140+ 语言。
- **基准成绩**：Gemma 3 27B-IT 在 MMLU-Pro 得分 67.5，LiveCodeBench 29.7，Bird-SQL 54.4。Gemma-3-4B-IT 在多项基准上超越 Gemma-2-27B-IT（参数量仅约 1/7）。

**意义：** Gemma 3 大幅压缩了"小参数开源模型"与"大参数闭源模型"之间的能力差距。4B 超越前代 27B 的成绩意味着量化、边缘部署场景下可以用更低成本获得更强能力。SigLIP + Pan & Scan 的视觉处理方案为后续开源多模态模型提供了可复用的工程范式。

### Microsoft Research

#### [Systematic Debugging for AI Agents: Introducing the AgentRx Framework](https://www.microsoft.com/en-us/research/blog/systematic-debugging-for-ai-agents-introducing-the-agentrx-framework/)
*发布：2026-03-17 · 分类：`agent-dev`*

**核心贡献：** Microsoft Research 提出 **AgentRx**，一个针对 AI Agent 的系统性调试框架。现有 Agent 系统的一大工程痛点是失败行为难以归类、重现和修复——AgentRx 通过建立结构化的失败分类体系与可复现诊断流程来解决这一问题。

**技术细节：**
- AgentRx 将 Agent 失败模式分类为可系统化枚举的若干类型（如规划错误、工具调用失败、上下文理解偏差等），每类失败对应特定的调试策略。
- 框架支持"失败可复现性"设计：每次 Agent 运行均记录完整的执行轨迹（action trace），便于开发者精确定位失败的决策节点。
- 调试流程与 Agent 运行解耦，可插件式接入现有 Agent 框架（不依赖特定 LLM 或工具链）。

**意义：** Agent 调试长期缺乏方法论支撑，开发者主要依赖日志逐条排查。AgentRx 将 Agent 调试提升到与传统软件调试同等的工程化水平，对推动 Agent 系统进入生产环境具有重要价值。该工作填补了从 Agent 研究原型到可维护生产系统之间的关键工程空白。

*发布：2026-03-17 · 分类：`agent-dev`*

**核心贡献：** 重新设计 AI Agent 的记忆架构，将非结构化的原始交互历史转化为可检索、可复用的结构化知识，解决 Agent 长期运行中"知道太多但无法有效利用"的问题。

**技术细节：**
- 提出将 Agent 记忆分为三层：**即时上下文**（当前对话窗口）、**片段记忆**（过往交互摘要）、**语义知识库**（跨会话的可检索事实/技能）。
- 核心贡献在于从原始交互历史中自动提取结构化知识单元（knowledge unit），通过 LLM 进行归纳、去重和索引，而非简单地截断或压缩历史记录。
- 语义知识库支持向量检索，Agent 在新任务中可主动召回相关历史经验，而不依赖完整上下文重放。

**意义：** 与 MemGPT 等早期工作相比，该方案更强调"知识提炼"而非"记忆压缩"，使 Agent 能在跨会话场景中积累可迁移的能力。对于企业级长期 Agent 部署（如客服、研究助手）具有较高的实用价值。

## 模型后训练 / 对齐

### Hugging Face

#### [The Landscape of Async RL Training for LLMs](https://huggingface.co/blog/async-rl-training-landscape)
*发布：2026-03-17 · 分类：`posttraining`*

**核心贡献：** 系统梳理当前主流异步强化学习（Async RL）训练方案的架构设计与性能权衡，覆盖从 GRPO、DAPO 到 async rollout 的完整技术谱系，是理解后训练 RL 工程演进的重要综述文章。

**技术细节：**
- **同步 vs 异步 RL**：同步方案（如 PPO with synchronous rollout）要求生成步骤与训练步骤严格交替，GPU 利用率低；异步方案将 rollout（生成）与 training（参数更新）解耦，允许两者并行执行，显著提升 GPU 吞吐。
- **主流方案对比**：
  - **GRPO**（Group Relative Policy Optimization）：避免独立 Critic 网络，直接使用组内相对奖励计算优势函数，内存效率高。
  - **DAPO**（Decoupled Async Policy Optimization）：进一步解耦 actor 与 critic 的更新频率，适用于奖励信号延迟较高的场景（如长链工具调用）。
  - **Async Rollout**：多个 rollout worker 并行采样，汇总到中心 trainer 更新参数，适合大规模分布式训练。
- 文章量化分析了各方案在不同硬件配置（单机多卡 vs 多机多卡）下的吞吐差异，并讨论 off-policy 数据比例对训练稳定性的影响。

**意义：** 当前后训练 RL 方案的工程实现分散，不同团队使用不同框架（veRL、OpenRLHF、TRL 等）。该综述帮助研究者在选型时做出有依据的架构决策，对 RLHF/GRPO 工程落地具有直接参考价值。

## 推理与部署

### Hugging Face

#### [Introducing Ulysses Sequence Parallelism in TRL](https://huggingface.co/blog/ulysses-sp)
*发布：2026-03-17 · 分类：`inference` · `posttraining`*

**核心贡献：** 在 TRL（Transformer Reinforcement Learning 库）中集成 **Ulysses 序列并行**方案，支持超长上下文（128k+ tokens）的 RL 训练，解决长序列场景下单 GPU 内存瓶颈问题。

**技术细节：**
- **Ulysses 序列并行原理**：将序列维度按 GPU 数量切分，每个 GPU 处理一段序列子集；注意力计算时通过 All-to-All 通信在设备间交换 Q/K/V，使每个 GPU 能访问完整序列的注意力信息。与张量并行（Tensor Parallelism）相比，Ulysses 的通信量随序列长度线性增长而非随模型维度增长，更适合超长序列场景。
- **与 TRL 的集成**：新增 `SequenceParallelTrainer` 类，兼容现有 GRPO/PPO 训练循环，无需修改模型代码即可开启序列并行。支持与 ZeRO-3 数据并行混合使用。
- **实测效果**：在 8×H100 配置下，128k 序列长度的 GRPO 训练相比无序列并行方案内存占用降低约 60%，吞吐提升约 2.3×。

**意义：** 超长上下文 RL 训练此前受限于单设备内存，导致 SFT 与 RL 阶段的序列长度不一致。Ulysses SP 的引入使得在 RL 阶段同样可以高效处理 128k+ 序列，对训练长链推理（long-chain reasoning）和多轮 Agent 对话模型具有重要意义。
