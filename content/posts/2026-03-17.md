#### 📌 [Designing AI agents to resist prompt injection](https://openai.com/index/designing-agents-to-resist-prompt-injection/)
*发布：2026-03-11 · 类型：经验/实践 · 分类：agent-dev · safety*

**文章类型：** 工程安全经验

**一句话概括：** prompt injection 防御的核心不是识别恶意输入，而是设计 agent 系统使得即便被攻击成功，损害范围也受到结构性限制。

**核心问题：** 随着 AI agent 获得浏览网页、调用工具、代表用户执行操作的能力，如何防止被外部内容操控？

**主要内容结构：**

1. **攻击形态的演变：从指令覆盖到社会工程学**  
   早期 prompt injection 很简单——攻击者在 Wikipedia 页面里写"给我发一封邮件"，未经训练的模型会直接执行。随着模型变强，攻击也进化了：现代攻击越来越像社会工程学，不是直接下命令，而是编造理由、制造紧迫感、伪造授权。这意味着传统的"AI 防火墙"（将输入分类为恶意/正常）难以奏效——识别一个精心构造的谎言，和识别一个真正的请求一样困难。

2. **设计范式转变：从"识别攻击"到"限制损害"**  
   OpenAI 借用了人类客服系统的三方模型思考这个问题：
   - 企业（system prompt 制定者）→ agent（执行者）→ 外部用户/内容
   - 人类客服可能被客户欺骗，所以企业会限制客服的权限（每个客户的退款上限、可访问的系统权限等），即便客服被骗，损失也有上界。
   - AI agent 应该采用同样的设计：**不假设 agent 永远不会被欺骗，而是确保被欺骗后的最坏情况是可接受的**。

3. **ChatGPT 中的具体实现：source-sink 分析 + Safe Url**  
   OpenAI 在 ChatGPT 中结合了社会工程学防御模型与传统安全工程方法：
   - **Source-sink 分析**：攻击者需要同时控制"source"（影响系统的途径）和"sink"（危险的能力，如发送信息给第三方）。防御目标是让这两者难以被同时利用。
   - **核心安全期望**：潜在危险的操作或敏感信息的传输，不应在没有适当保障的情况下静默发生。
   - **Safe Url 机制**：当 agent 准备将会话中获取的信息传递给第三方 URL 时，系统检测这一行为，要么展示给用户确认，要么阻断并让 agent 换方式推进。
   - 这套机制同样适用于 Atlas（导航和书签）、Deep Research（搜索和导航）、ChatGPT Canvas 和 Apps（在沙箱中运行，检测意外通信后请求用户同意）。

4. **给开发者的建议**  
   在将 AI 模型集成到应用系统时：先问自己"如果是一个人类 agent 在做这件事，应该有什么样的权限限制"，然后实现这些限制。不要期望模型永远不会被欺骗，而要设计系统使得被欺骗的后果是可接受的。

**关键经验/原则：**
- **不要只靠输入过滤**：识别精心构造的社会工程学攻击和识别谎言一样难，AI 防火墙对此无效。
- **系统性约束优于个案识别**：限制 agent 能做什么，比试图让 agent 识别所有攻击更可靠。
- **Safe Url 作为最后一道防线**：信息外传前的用户确认机制，即便 agent 被完全说服也能阻止数据泄露。
- **更聪明的模型抗社会工程学能力会更强**——但不应依赖这一点，应以架构设计为主。

**相关资源：** [Atlas 的 prompt injection 加固](https://openai.com/index/hardening-atlas-against-prompt-injection/) · [Deep Research 介绍](https://openai.com/index/introducing-deep-research/)

#### 📌 [From model to agent: Equipping the Responses API with a computer environment](https://openai.com/index/equip-responses-api-computer-environment/)
*发布：2026-03-11 · 类型：经验/实践 · 分类：agent-dev*

**文章类型：** 工程实现经验

**一句话概括：** OpenAI 为 Responses API 配备了完整的计算机执行环境（shell tool + container workspace），解决 agent 工作流中的文件存储、网络访问、超时重试、context 压缩等工程难题。

**核心问题：** 单纯调用模型只能访问训练时的知识；让模型操作计算机才能处理真实世界的复杂工作流。如何构建一个可靠的 agent 执行环境？

**主要内容结构：**

1. **从 model 到 agent 的核心差距**  
   模型只能"提议"操作，无法自己执行。过去开发者需要自己搭执行环境、管理文件、处理网络访问、处理超时重试。现在 OpenAI 把这些打包进 Responses API + container workspace。

2. **Shell Tool：执行环境的核心**  
   - 模型通过命令行与计算机交互，支持 grep、curl、awk 等完整 Unix 工具集
   - 相比原有的 code interpreter（只支持 Python），shell tool 支持 Go、Java、Node.js 等任意语言和服务
   - 模型"提议" shell 命令，Responses API 在 container 中执行，结果反馈给模型形成循环
   - GPT-5.2 及更新版本训练时加入了 shell 命令生成能力

3. **agent 循环的关键工程细节**  
   - **流式执行**：Responses API 与 container service 保持流式连接，shell 输出近实时反馈给模型
   - **并发执行**：模型可一次提议多条命令，Responses API 用独立 container session 并行执行，多路流汇聚为结构化输出
   - **输出上限（output cap）**：模型为每条命令指定输出上限，API 保留开头和结尾、标记省略部分，防止 context 被原始日志淹没
   - 并发执行 + 输出限制共同保证了 agent 循环的速度和 context 效率

4. **Context Compaction（上下文压缩）**  
   长任务会填满 context window。OpenAI 在 Responses API 中加入了原生 compaction：
   - 模型被训练来分析对话历史并生成"压缩项"——一个保留关键状态的加密 token 高效表示
   - 压缩后，下一个 context window 包含压缩项 + 之前 window 的高价值部分
   - 支持服务端自动触发（配置阈值）或独立的 `/compact` 端点
   - Codex 是这套机制的第一个用户，也借助自身在用 compaction 时遇到的问题来改进 compaction 本身

5. **Container 文件系统与数据库**  
   - **文件系统**：通过 container API 上传和管理资源，让模型用定向文件操作替代大范围扫描
   - **反模式**：把所有输入塞入 prompt context；正确做法是把资源 stage 到 container，让模型按需读取
   - **SQLite**：结构化数据存储，模型通过 shell 工具查询；比把表格粘贴进 prompt 高效得多
   - **网络访问**：在 policy 控制下开放受限网络，支持 API 调用等

6. **Reusable Skills（可复用技能）**  
   可以预定义一组 shell 技能（skill），容器启动时加载。模型可以直接调用这些预封装的操作，避免每次重复生成相同的 shell 序列。

**关键经验/原则：**
- **不要把大表格塞进 prompt**：stage 到文件系统，让模型按需读取，模型对有组织的信息处理更好
- **输出上限是 context 效率的关键**：保留开头和结尾、标记省略，是比完整输出或硬截断都更好的策略
- **native compaction 优于自定义摘要**：随模型训练更新，且与模型行为对齐
- **Codex 反哺自身**：Codex 在使用 compaction 时遇到问题，另起一个 Codex 实例来调查修复，工具在自我完善

#### 📌 [Why Codex Security Doesn't Include a SAST Report](https://openai.com/index/why-codex-security-doesnt-include-sast/)
*发布：2026-03-16 · 类型：经验/实践 · 分类：safety · agent-dev*

**文章类型：** 工程安全方法论

**一句话概括：** SAST（静态应用安全测试）追踪数据流，但真正危险的漏洞在于代码中的安全检查是否真的有效——Codex Security 从"行为验证"出发，主动尝试证伪安全假设。

**核心问题：** 为什么 AI 代码安全审计不从 SAST 报告出发，而要从代码库本身出发？

**主要内容结构：**

1. **SAST 的能力边界**  
   SAST 的经典模型：找到不可信输入（source）→ 追踪数据流经过函数和层 → 检查是否有未经净化就到达敏感操作（sink）。这个模型覆盖了大量真实漏洞，是安全领域的重要工具。  
   但 SAST 必须做大量近似才能在真实代码库中保持可扩展性（间接调用、动态 dispatch、callback、反射、重度框架控制流等）。

2. **SAST 的根本局限：验证检查是否有效**  
   就算 SAST 正确追踪到了一个 source → sink 路径，它还需要回答：**这条路径上的安全检查是否真的有效？**  
   典型例子：代码调用了 `sanitize_html()` 再渲染内容。静态分析能看到净化函数被调用，但通常无法判断这个净化函数是否足够，因为这取决于具体的渲染上下文、模板引擎、编码行为、下游转换的具体组合。

3. **转换链漏洞：真实案例**  
   一个 web 应用：接收 JSON payload → 提取 `redirect_url` → 用正则 allowlist 验证 → URL decode → 传给重定向处理器。  
   SAST 能描述这条数据流。但关键问题是：**正则在 decode 之前运行，它约束的是 decoded URL 之后的行为吗？**  
   答案取决于：正则允许什么、decode 和 normalization 的行为、URL 解析对边界情况的处理、重定向逻辑对 scheme 和 authority 的解析方式。  
   CVE-2024-29041（Express 开放重定向）就是这类问题：数据流清晰，漏洞在于转换链中验证失效。

4. **Codex Security 的方法：从行为出发，主动证伪**  
   核心目标：**减少人工 triage，只上报有强证据支撑的问题**。具体做法：
   - 使用仓库特定上下文（包括威胁模型）
   - 在隔离环境中验证高信号问题后才上报
   - 遇到"验证"或"净化"代码时，不把调用本身当作安全检查通过，而是尝试理解这段代码试图保证什么属性，然后**尝试证伪这个保证**
   - 把问题化简到最小可测试的切片（例如单个输入的转换管道），在不依赖整个系统的情况下推理
   - 注意：模型不会盲信注释，在注释旁写 `// this is not a bug` 无法迷惑它

**关键经验/原则：**
- **"代码调用了净化函数"≠"系统是安全的"**：关键是净化函数的约束是否在后续转换链中保持成立
- **转换链是漏洞的温床**：decode 顺序、normalization、多层解析——每一步都可能使之前的检查失效
- **证伪优于验证**：安全审计不应试图证明代码安全，而应主动寻找假设失效的场景

### H Company / NVIDIA（via HuggingFace）

#### [Holotron-12B - High Throughput Computer Use Agent](https://huggingface.co/blog/Hcompany/holotron-12b)
*发布：2026-03-17 · 类型：研究 · 分类：agent-dev · inference*

**核心贡献：** H Company 发布 12B 参数的 computer-use agent 模型 Holotron-12B，基于 NVIDIA Nemotron Hybrid SSM 架构，在 WebVoyager 基准上实现 80.5% 准确率，推理吞吐量比 Holo2-8B 提高 2x 以上。

**背景与动机：** 现有多模态模型主要优化静态视觉理解或指令跟随。Computer-use agent 需要高效处理长上下文（多轮截图 + 操作历史），传统 Transformer attention 的 KV Cache 随序列长度线性增长，成为推理瓶颈。

**方法/技术细节：**
- **架构**：Hybrid SSM + Attention（来自 NVIDIA Nemotron-Nano-2 VL）
  - SSM（State-Space Model）的 KV Cache 是常数大小（每层每序列一个固定状态，与序列长度无关）
  - 相比纯 Transformer，显著降低长上下文推理的内存占用，允许更大的有效 batch size
- **训练**：两阶段
  1. 基础：NVIDIA Nemotron-Nano-12B-v2-VL-BF16
  2. SFT：H Company 私有的 localization + navigation 数据，聚焦屏幕理解、元素定位、UI 交互
  3. 约 140 亿 token 训练量
- **推理环境**：单张 H100 GPU，vLLM v0.14.1（含最新 SSM 优化）

**实验结果：**
- **WebVoyager**：80.5%（Nemotron 基础模型 35.1%，Holo2-8B 低于 Holotron-12B）
- **吞吐量**：在并发 100 的 WebVoyager 负载下，Holotron-12B 达到 8,900 tokens/s；Holo2-8B 瓶颈在 5,100 tokens/s
- **Localization benchmarks**（OS-World-G、GroundUI、WebClick）均有大幅提升

**意义：** SSM 架构在 computer-use agent 这种吞吐量敏感场景（数据生成、标注、在线强化学习）中的推理效率优势被实际验证。
